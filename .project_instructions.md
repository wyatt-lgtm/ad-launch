## Post Import / Visibility Reconciliation (May 2026)

### Root cause fixed
`handleProgressComplete` used stale React state (`posts`) to verify import success ŌĆö React hadn't re-rendered yet after `fetchPosts()` called `setPosts()`. Now uses `pollResult.imported` count from the API response + a fresh verification fetch.

### What changed
- **`handleProgressComplete`** (`social-dashboard.tsx`) ŌĆö no longer uses stale `posts` state. Uses poll API `imported` count, then does a fresh fetch to `/api/social/posts` to verify the imported post is visible in the current business queue. If hidden by status filter or wrong business, shows specific diagnostic error and auto-clears filter.
- **`/api/social/missions/poll`** ŌĆö enhanced import validation: validates required fields (caption, imageUrl, cta, source attribution), assigns `generation_incomplete` status for posts with image-only (no caption), returns `importedComplete`/`importedIncomplete` counts + `diagnostics` array with `missingFields`, `importError` per incomplete post.
- **`generation_incomplete` status** ŌĆö new status type added to STATUS_CONFIG, filter buttons, and import logic. Posts with partial output (e.g., image but no caption) get this status instead of silently being `pending_approval`.
- **Diagnostic card** ŌĆö `generation_failed` and `generation_incomplete` posts show an expanded diagnostic banner with: workflow ID, task ID, run ID, business name, source article, missing fields, and "Retry Import" / "View Workflow" buttons.
- **Frontend rendering guards** ŌĆö normal posts show inline warnings for: missing caption, missing source attribution, missing workflowId link.
- **Retry import** ŌĆö `retryImportForPost()` deletes the failed shell record, re-polls Tombstone, and refreshes posts.
- **Poll status message** ŌĆö now shows "(X incomplete/failed)" suffix when incomplete imports exist.
- **Regression tests** (`__tests__/post-import-reconciliation.test.ts`) ŌĆö 15 tests covering import validation, queue count consistency, malformed post visibility, business mismatch, post-import verification, and contradictory message prevention.

## Workflow ID Persistence Fix (May 2026)

### Root cause fixed
When generating social posts via `businessId` (rather than `analysisId`), the generate route resolved `websiteUrl` from the Business record but never looked up the associated Analysis. This left `resolvedAnalysisId` as null, so the `socialMissionId` (containing workflow IDs) was never stored on the Analysis record. The poll route uses `analysis.socialMissionId` to collect workflow IDs for querying the Tombstone content queue, so these workflows were invisible to polling ŌĆö posts generated successfully but never imported.

### What changed
- **`/api/social/generate/route.ts`** ŌĆö after finding the Business record via `businessId`, if `resolvedAnalysisId` is still null, now looks up the most recent Analysis for that business and user. This ensures `socialMissionId` gets written even for business-based generations (not just analysis-based ones).
- **`handleProgressComplete` retry logic** (`social-dashboard.tsx`) ŌĆö added retry mechanism (5 attempts, 15-second intervals) when the first poll returns 0 imports after workflow completion. The Tombstone content queue can take time to be updated after workflow finishes.
- **Data fix** ŌĆö manually patched North Shore Tavern's analysis record with 3 missing workflow IDs that were lost before the code fix.

## Draft-to-Post Enhancements (June 2026)

### Bug fix: Create-from-draft business assignment

- Poll route (`missions/poll/route.ts`) now falls back to Tombstone `/tasks/{id}` API to get `workflow_id` when content queue entry lacks it.

### Article URL input (new feature)
- Draft form now has an **Article URL** input field ŌĆö paste any web article URL
- Backend fetches article via `fetchAndParseArticle()`, builds carousel via `buildCarouselPackage()`, generates slide images via `generateCarouselImages()`, and creates SocialPost directly (no Tombstone polish needed)
- When both article URL and draft text are provided, draft text is prepended as "user angle/context" to the article body
- Response includes `{ immediate: true, postId }` ŌĆö frontend refreshes post list immediately

### Carousel format for user-written drafts (new feature)
- Draft form has **Carousel Format** checkbox (auto-checked + disabled when article URL is present)
- When checked, `buildCarouselFromDraft()` sends draft text to LLM to detect list/event/tip content suitable for multi-slide format
- If LLM determines content IS carousel-worthy ŌåÆ generates slides + creates SocialPost directly
- If LLM determines content is NOT carousel-worthy ŌåÆ falls through to standard Tombstone polish pipeline (graceful degradation)
- Source type `user_draft_carousel` distinguishes these from article-based carousels

### Three paths in create-from-draft
1. **PATH 1**: `articleUrl` ŌåÆ article carousel (immediate)
2. **PATH 2**: `draftText` + `forceCarousel` ŌåÆ draft carousel if worthy, else falls to PATH 3 (immediate if carousel)
3. **PATH 3**: Standard draft ŌåÆ Tombstone polish mission (existing behavior)

## Identity Lock for Lane Missions (June 2026)

### Root cause fixed
`createLaneMission()` in `lib/tombstone.ts` was the ONLY mission creation function that lacked the `buildIdentityLockBlock()`. This meant the 3-lane analysis launch (website/news/holiday) sent commands to Tombstone WITHOUT an advertiser identity lock. When Tombstone agents processed these commands, they could pick up identity from adjacent/recent missions for other businesses (e.g., Blazin Hog CTA appearing on West Highlands Auto Repair ads).

### What changed
- **`createLaneMission()`** ŌĆö now accepts optional `businessName` parameter and prepends `buildIdentityLockBlock()` to ALL lane commands (website, news, holiday), matching the pattern used by `createSocialMissions()`, `createSingleStoryPost()`, and `createWeeklyTipPost()`.
- **`confirm-and-launch/route.ts`** ŌĆö passes `businessName` to all 3 `createLaneMission()` calls.
- **`generate-more/route.ts`** ŌĆö passes `businessName` to `createLaneMission()` call.
- **Data fix** ŌĆö corrected contaminated CTA on West Highlands Auto Repair analysis (was "Check your address at blazinghog.co", now "Schedule Your Service").

### Secondary Text Lines in Image Renders (June 2026)
- **Problem**: When users write drafts with specific data (dates, schedules, stats), those facts only appeared in the caption, not the rendered image. Users expected key data like game dates to be visible in the creative itself.
- **Fix**: Added `secondary_text_lines` support across the full pipeline:
  - `createDraftPolishMission()` in `lib/tombstone.ts` now instructs the pipeline to extract key data points as secondary text
  - Don Draper's creative direction JSON schema includes `secondary_text_lines` field
  - Andy Warhol's `_build_full_comp_prompt()` renders secondary text lines between headline and CTA
  - Both multi-campaign and single-campaign render paths pass `secondary_text_lines` through
- **Also**: Added instruction for people/subjects to face TOWARD the action (e.g., fans facing TV, not away from it)
