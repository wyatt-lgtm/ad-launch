# Ad Launch - Project Instructions

## Post-Production QA Gate (added May 2026)
- `lib/post-qa.ts` — LLM vision QA for social post images, called from `completion-check/route.ts`
- Runs AFTER Tombstone workflow completes, BEFORE promoting PostPackage to `ready`
- 6 hard-fail rules: blank space, CTA mismatch, subject relevance, text integrity, business connection, stock imagery
- On QA failure: status → `rejected`, reasons stored in `qaRejectReason` field, no credit charge, no email
- On LLM failure: pass-through (score=-1) so pipeline doesn't block on transient errors
- Uses `gpt-5.4-mini` with vision via Abacus AI LLM API

## Credit Enforcement
- **CREDIT_ENFORCEMENT_ENABLED = false** in `lib/credits.ts` (added May 2026)
- All credit code, UI, tracking, and infrastructure remains intact — only enforcement is suspended
- `checkCredits()` always returns `allowed: true` when flag is false
- `chargeForPostPackage()` skips `INSUFFICIENT_CREDITS` throw when flag is false (still records transactions if credits exist)
- Frontend warning blocks and disabled buttons are wrapped with `false as boolean &&` to suppress without removing
- To re-enable at product launch: flip `CREDIT_ENFORCEMENT_ENABLED` to `true` in `lib/credits.ts`

## Business Context Persistence (May 2026)
- `useActiveBusiness` hook uses `localStorage` (key: `adlaunch_active_business_id`) for cross-tab/refresh persistence
- Also writes to `sessionStorage` for backward compat
- Cross-tab sync via `useSyncExternalStore` + `storage` event listener
- Dashboard sets the business ID in localStorage when user clicks a business card
- Social Posts, Publish Queue, Content Feeds, Credits all use `useActiveBusiness` hook
- Social Posts API (`/api/social/posts`) filters by `businessId` query param
- Content Queue API (`/api/content/queue`) filters analyses by `businessId` before extracting Tombstone workflow IDs
- Publishing Dashboard now has business picker, banner, and `businessId` filtering (matches Social Posts pattern)
- Stale data guard: when `activeBusinessId` changes, posts are cleared immediately before refetch
- Diagnostic logging: `[BusinessContext]` for selection changes, `[SocialPosts]` for post loads
