# Ad Launch - Project Instructions

## Overview
Ad Launch is a lead-gen platform that takes a business URL, performs deep research via Tombstone OS (FastAPI backend), and generates visual Facebook ads, SEO audits, Google Ads copy, website concepts, budget recommendations, and 90-day posting plans.

## Architecture
- **Frontend:** Next.js 14 (App Router) with Tailwind CSS, deployed at ad-launch-1nfyr8.abacusai.app + connect.launchmarketing.com
- **Backend Pipeline:** Tombstone OS (FastAPI on Render) — multi-agent system for research + creative generation
- **Database:** PostgreSQL (Prisma ORM)
- **Auth:** NextAuth with credentials (business email only)
- **CRM:** GoHighLevel (GHL) for contact management + confirmation emails
- **Image Storage:** S3 (AWS) for gpt-image-1 generated ad images; R2 (Cloudflare) for legacy Tombstone artifacts
- **GitHub Repos** (May 2026 separation):
  - `wyatt-lgtm/ad-launch` (branch: `main`) — Ad Launch frontend (this project)
  - `wyatt-lgtm/tombstone` (branch: `main`) — Tombstone OS backend (separate Render deploy)
  - `.env` stripped from git history in both repos

### Starter Credits (Auto-Grant)
- Every new beta business automatically receives **6 starter credits** once via `grantStarterCredits()`.
- Idempotency key: `starter-grant:{businessId}` — prevents double grants.
- Triggered in both `/api/analyze` and `/api/register` on business upsert (non-fatal on error).
- `CreditAccount` created with `creditPlanName: 'beta'`, `creditStatus: 'active'`.
- Admin backfill: `POST /api/admin/credits/grant-starter` — grants to all businesses missing starter grant, idempotent.
- `starter_grant` is a valid `TransactionType` in `lib/credits.ts`.

### Credit Lot / Expiration System (May 2026)
- **CreditLot model**: per-grant credit bucket tracking `originalAmount`, `remainingAmount`, `expiresAt`, `closureExpiresAt`, `creditType` (starter/monthly/recharge/admin/refund)
- **Expiration rules**:
  - Starter/monthly credits: expire 60 days after grant (`GRANT_EXPIRY_DAYS = 60`)
  - Recharge credits: never expire while account active; expire 30 days after account closure
  - Admin credits: optionally expire if `expiresAt` supplied
  - Refund credits: inherit expiration from original source lots
- **FIFO consumption**: `consumeLotsInTransaction()` drains earliest-expiring lots first, non-expiring last
- **Expiration sweep**: `expireCreditsForBusiness()` — finds lots past effective expiration, zeroes remainingAmount, creates `credit_expired` transactions, adjusts CreditAccount balance
- **Account closure**: `closeAccountCreditWindow(businessId)` — sets `accountClosedAt` + `creditClosureExpiresAt` on CreditAccount, marks recharge lots with 30-day `closureExpiresAt`
- **Balance API** (`/api/credits/balance`): now returns `expiringSoonCredits`, `nextExpirationDate`, `accountClosedAt`, `creditClosureExpiresAt`, `expirationPolicy`
- **UI updates**: CreditBadge shows "X credits expire in Y days" warning; BillingSection shows expiring-soon row with clock icon
- **Admin endpoints**:
  - `GET /api/admin/credits/lots?businessId=&includeEmpty=` — list credit lots
  - `POST /api/admin/credits/expire-now` — force-expire all eligible lots for a business
  - `POST /api/admin/credits/close-account-credit-window` — trigger 30-day closure window
- **Constants**: `EXPIRING_SOON_DAYS = 14` (threshold for UI warnings)
- **Billing integration**: `handleSubscriptionDeleted()` in `lib/billing.ts` calls `closeAccountCreditWindow()` automatically

## Scout-to-Post Email Workflow (May 2026)

### Overview
- MVP email-driven workflow: daily scout emails → one-click "Create Post" magic links → Tombstone generates 1 post → completion email with review/download links
- NO publishing integrations — manual posting only (copy text, download image, download package)

### New Models
- `ScoutReport` — tracks each daily email send (businessId, userId, status, storiesCount, sentAt, expiresAt)
- `ScoutStory` — individual stories within a report (title, source, sourceUrl, sourceType, summary, relevance, suggestedAngle)
- `MagicToken` — signed, expiring (72h), single-use tokens for email actions (create_post, review_stories)
- `PostPackage` — output of a Tombstone single-story workflow (postCopy, headline, cta, hashtags, imageUrl, status: generating→ready→downloaded→draft→rejected→posted)

### Magic Links
- HMAC-SHA256 signed with NEXTAUTH_SECRET, base64url-encoded JSON payload
- DB-backed single-use enforcement for cost-incurring actions (create_post)
- 72h expiry, nonce for uniqueness
- Lib: `lib/magic-token.ts` — `generateMagicToken()`, `verifyMagicToken()`

### API Routes
- `GET /api/scout/create-post?token=...` — magic link handler: validates token, creates PostPackage, launches Tombstone, redirects to /scout/confirm
- `POST /api/scout/completion-check` — admin endpoint: polls generating PostPackages, checks Tombstone status, sends completion email (NOTIF_ID_POST_READY)
- `GET /api/post-package/[id]` — post package details (session or magic token auth)
- `POST /api/post-package/[id]/action` — save_draft/reject/mark_posted (session auth)
- `GET /api/post-package/[id]/download` — download package as JSON
- `GET /api/scout/report/[id]` — scout report with stories (session or magic token auth)
- `POST /api/scout/create-posts-batch` — authenticated batch create (1-3 stories from review page)

### Pages
- `/scout/confirm` — confirmation page after magic link click (success/active_workflow/error states)
- `/scout/expired` — expired/invalid token page with reason-specific messages
- `/post/[id]` — full post review/download page (Copy Text, Download Image, Download Package, Save Draft, Reject, Mark as Manually Posted)
- `/scout/review/[id]` — review all stories page with checkboxes (max 3), grouped by type

- Generates per-story magic tokens (create_post action)
- Generates review-all magic token (review_stories action)
- Email HTML includes per-story "✨ Create Post" buttons and "📋 Review All Stories" button

- `createScoutStoryMission()` in `lib/tombstone.ts` — builds single-story command with source: daily_scout_email, workflow_type: single_story_post, max_posts: 1
- Uses existing `sendCommand()` entry point
