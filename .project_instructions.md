## Post Import / Visibility Reconciliation (May 2026)

### Root cause fixed
`handleProgressComplete` used stale React state (`posts`) to verify import success ‚Äî React hadn't re-rendered yet after `fetchPosts()` called `setPosts()`. Now uses `pollResult.imported` count from the API response + a fresh verification fetch.

### What changed
- **`handleProgressComplete`** (`social-dashboard.tsx`) ‚Äî no longer uses stale `posts` state. Uses poll API `imported` count, then does a fresh fetch to `/api/social/posts` to verify the imported post is visible in the current business queue. If hidden by status filter or wrong business, shows specific diagnostic error and auto-clears filter.
- **`/api/social/missions/poll`** ‚Äî enhanced import validation: validates required fields (caption, imageUrl, cta, source attribution), assigns `generation_incomplete` status for posts with image-only (no caption), returns `importedComplete`/`importedIncomplete` counts + `diagnostics` array with `missingFields`, `importError` per incomplete post.
- **`generation_incomplete` status** ‚Äî new status type added to STATUS_CONFIG, filter buttons, and import logic. Posts with partial output (e.g., image but no caption) get this status instead of silently being `pending_approval`.
- **Diagnostic card** ‚Äî `generation_failed` and `generation_incomplete` posts show an expanded diagnostic banner with: workflow ID, task ID, run ID, business name, source article, missing fields, and "Retry Import" / "View Workflow" buttons.
- **Frontend rendering guards** ‚Äî normal posts show inline warnings for: missing caption, missing source attribution, missing workflowId link.
- **Retry import** ‚Äî `retryImportForPost()` deletes the failed shell record, re-polls Tombstone, and refreshes posts.
- **Poll status message** ‚Äî now shows "(X incomplete/failed)" suffix when incomplete imports exist.
- **Regression tests** (`__tests__/post-import-reconciliation.test.ts`) ‚Äî 15 tests covering import validation, queue count consistency, malformed post visibility, business mismatch, post-import verification, and contradictory message prevention.

## Workflow ID Persistence Fix (May 2026)

### Root cause fixed
When generating social posts via `businessId` (rather than `analysisId`), the generate route resolved `websiteUrl` from the Business record but never looked up the associated Analysis. This left `resolvedAnalysisId` as null, so the `socialMissionId` (containing workflow IDs) was never stored on the Analysis record. The poll route uses `analysis.socialMissionId` to collect workflow IDs for querying the Tombstone content queue, so these workflows were invisible to polling ‚Äî posts generated successfully but never imported.

### What changed
- **`/api/social/generate/route.ts`** ‚Äî after finding the Business record via `businessId`, if `resolvedAnalysisId` is still null, now looks up the most recent Analysis for that business and user. This ensures `socialMissionId` gets written even for business-based generations (not just analysis-based ones).
- **`handleProgressComplete` retry logic** (`social-dashboard.tsx`) ‚Äî added retry mechanism (5 attempts, 15-second intervals) when the first poll returns 0 imports after workflow completion. The Tombstone content queue can take time to be updated after workflow finishes.
- **Data fix** ‚Äî manually patched North Shore Tavern's analysis record with 3 missing workflow IDs that were lost before the code fix.

## Draft-to-Post Enhancements (June 2026)

### Bug fix: Create-from-draft business assignment
- `create-from-draft/route.ts` was storing workflows on the wrong business's analysis (most recent analysis instead of selected business). Fixed to look up analysis by `businessId`.
- `create-from-draft/route.ts` now creates a `GenerationRun` record so poll route can map back to the workflow.
- Poll route (`missions/poll/route.ts`) now falls back to Tombstone `/tasks/{id}` API to get `workflow_id` when content queue entry lacks it.

### Article URL input (new feature)
- Draft form now has an **Article URL** input field ‚Äî paste any web article URL
- Backend fetches article via `fetchAndParseArticle()`, builds carousel via `buildCarouselPackage()`, generates slide images via `generateCarouselImages()`, and creates SocialPost directly (no Tombstone polish needed)
- When both article URL and draft text are provided, draft text is prepended as "user angle/context" to the article body
- Response includes `{ immediate: true, postId }` ‚Äî frontend refreshes post list immediately

### Carousel format for user-written drafts (new feature)
- Draft form has **Carousel Format** checkbox (auto-checked + disabled when article URL is present)
- When checked, `buildCarouselFromDraft()` sends draft text to LLM to detect list/event/tip content suitable for multi-slide format
- If LLM determines content IS carousel-worthy ‚Üí generates slides + creates SocialPost directly
- If LLM determines content is NOT carousel-worthy ‚Üí falls through to standard Tombstone polish pipeline (graceful degradation)
- Source type `user_draft_carousel` distinguishes these from article-based carousels

### Three paths in create-from-draft
1. **PATH 1**: `articleUrl` ‚Üí article carousel (immediate)
2. **PATH 2**: `draftText` + `forceCarousel` ‚Üí draft carousel if worthy, else falls to PATH 3 (immediate if carousel)
3. **PATH 3**: Standard draft ‚Üí Tombstone polish mission (existing behavior)
