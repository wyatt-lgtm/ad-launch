## Post Import / Visibility Reconciliation (May 2026)

### Root cause fixed
`handleProgressComplete` used stale React state (`posts`) to verify import success — React hadn't re-rendered yet after `fetchPosts()` called `setPosts()`. Now uses `pollResult.imported` count from the API response + a fresh verification fetch.

### What changed
- **`handleProgressComplete`** (`social-dashboard.tsx`) — no longer uses stale `posts` state. Uses poll API `imported` count, then does a fresh fetch to `/api/social/posts` to verify the imported post is visible in the current business queue. If hidden by status filter or wrong business, shows specific diagnostic error and auto-clears filter.
- **`/api/social/missions/poll`** — enhanced import validation: validates required fields (caption, imageUrl, cta, source attribution), assigns `generation_incomplete` status for posts with image-only (no caption), returns `importedComplete`/`importedIncomplete` counts + `diagnostics` array with `missingFields`, `importError` per incomplete post.
- **`generation_incomplete` status** — new status type added to STATUS_CONFIG, filter buttons, and import logic. Posts with partial output (e.g., image but no caption) get this status instead of silently being `pending_approval`.
- **Diagnostic card** — `generation_failed` and `generation_incomplete` posts show an expanded diagnostic banner with: workflow ID, task ID, run ID, business name, source article, missing fields, and "Retry Import" / "View Workflow" buttons.
- **Frontend rendering guards** — normal posts show inline warnings for: missing caption, missing source attribution, missing workflowId link.
- **Retry import** — `retryImportForPost()` deletes the failed shell record, re-polls Tombstone, and refreshes posts.
- **Poll status message** — now shows "(X incomplete/failed)" suffix when incomplete imports exist.
- **Regression tests** (`__tests__/post-import-reconciliation.test.ts`) — 15 tests covering import validation, queue count consistency, malformed post visibility, business mismatch, post-import verification, and contradictory message prevention.

## Workflow ID Persistence Fix (May 2026)

### Root cause fixed
When generating social posts via `businessId` (rather than `analysisId`), the generate route resolved `websiteUrl` from the Business record but never looked up the associated Analysis. This left `resolvedAnalysisId` as null, so the `socialMissionId` (containing workflow IDs) was never stored on the Analysis record. The poll route uses `analysis.socialMissionId` to collect workflow IDs for querying the Tombstone content queue, so these workflows were invisible to polling — posts generated successfully but never imported.

### What changed
- **`/api/social/generate/route.ts`** — after finding the Business record via `businessId`, if `resolvedAnalysisId` is still null, now looks up the most recent Analysis for that business and user. This ensures `socialMissionId` gets written even for business-based generations (not just analysis-based ones).
- **`handleProgressComplete` retry logic** (`social-dashboard.tsx`) — added retry mechanism (5 attempts, 15-second intervals) when the first poll returns 0 imports after workflow completion. The Tombstone content queue can take time to be updated after workflow finishes.
- **Data fix** — manually patched North Shore Tavern's analysis record with 3 missing workflow IDs that were lost before the code fix.
